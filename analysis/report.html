Cosmos DB Hierarchical Partition Keys — Go vs .NET deep analysis

29 query configs profiled · concurrency sweep 1–6 · depth = number of HPK levels constrained. RU is server-computed, but not identical across SDKs — Go runs ~10–19% higher than .NET for the same query (see §B); latency mixes transport (.NET Direct/TCP vs Go Gateway/HTTPS). Superscripts link to references.

Abstract. This report benchmarks Azure Cosmos DB hierarchical partition keys (HPK, /year/month/day) on a 1.27M-document container spread across 10 physical partitions, comparing the Go (azcosmos) and .NET (Microsoft.Azure.Cosmos) SDKs. It sweeps a matrix of prefix depth (3 / 2 / 1 / 0 HPK levels) × query expression (WHERE clause, explicit partition key, or both) × result form (SELECT * / COUNT), capturing request charge, latency, bytes read, documents scanned and physical partitions contacted, plus a client-concurrency sweep. Findings: deeper prefixes prune to fewer partitions (10→1), and RU scales ~657× by depth. The two SDKs report RU of the same order of magnitude for a given query, but not identical — Go runs ~10–19% higher than .NET (see §B) — and they diverge further in execution — .NET resolves partitions client-side and reads them in parallel over TCP (Direct mode), while Go routes and serves entirely through the gateway, making it slower on large cross-partition scans and unable to run cross-partition aggregates. The partial-PartitionKey over-read (#5404) is also reproduced. Methodology, per-query graphs and full data follow; the exhaustive tables are in the appendix.

A · Architecture — how a partial-HPK query routes, and why the SDKs differ

Where that routing decision is made differs between the two SDKs. When a query constrains a prefix of the hierarchical key (only year, or year+month), Cosmos can restrict execution to the physical partitions whose hash-range overlaps that prefix, rather than fan out to all of them[1].

Go — gateway routes and serves. azcosmos is gateway-only for queries; Direct/TCP mode exists only on the .NET and Java SDKs[3]. The client passes an empty partition key plus the WHERE clause; the gateway parses it, prunes to the matching partition-key ranges, executes there, and pages results back — one round-trip per page. Cross-partition aggregates are refused by the gateway[6]. The ranges the gateway returned (captured from the x-ms-documentdb-partitionkeyrangeid response header per page):

.NET — client routes, partitions serve. In Direct mode (default) the client caches the partition-key-range map, computes the target ranges itself from the WHERE prefix, opens TCP (rntbd) connections directly to the partition replicas, and queries them in parallel; a client-side pipeline merges/aggregates (so cross-partition COUNT/DISTINCT work)[4]. The gateway is used only for metadata. The .NET trace for the same 1-HPK query confirms this:

  • client-side routing stages: Get Partition Key Range Cache, Get Partition Key Ranges
  • direct TCP transport: rntbd://cdb-ms-prod-australiaeast1-be60.documents.azure.com:14049
  • partition replica(s) read directly: f839c82e-1e60-4efc-a90c-81361e039f52

Consequence (see §4): .NET's parallel-direct execution is faster on large scans, and it can run cross-partition aggregates; Go is simpler but gateway-bottlenecked (one round-trip per page). RU is server-computed and the same order of magnitude across SDKs, but not identical — Go runs ~10–19% higher than .NET (see §B)[9].

B · Go vs .NET SDK — detailed differences

Both SDKs target the same Cosmos DB NoSQL API and the same HPK container, and both return RU of the same order of magnitude for a given query (RU is computed server-side) — but not identical: Go runs ~10–19% higher than .NET for the same query, widening on larger scans (see below). They use different execution models, and that difference accounts for the behavioural, performance, and RU differences measured here.

Measured consequences (this run)

  • Large cross-partition scans run substantially faster on .NET. 0-HPK SELECT * (1.27M docs) p50 at concurrency 6: Go 441 s vs .NET 99.8 s. Go makes roughly 1,300 sequential gateway round-trips (one per page); .NET queries all 10 partitions in parallel over TCP.
  • Go cannot run cross-partition aggregates. Any COUNT/DISTINCT not scoped to a single full partition key fails with BadRequest — including a 2-HPK COUNT whose data physically lives on one partition, because Go can’t pass a partial partition key, so the gateway treats it as cross-partition.
  • RU is server-computed but not identical — Go runs ~10–19% higher than .NET for the same query, and the gap widens with scan size: 1-HPK SELECT * = 18,434 RU (Go) vs 16,574 RU (.NET) (+11.2%); 3-HPK = 33 vs 30 RU (+9.7%); the gap reaches +18.7% on the 0-HPK full scan (23,568 vs 19,859 RU). The likely cause is execution path, not transport: .NET's Direct-mode + Optimistic Direct Execution (§C.3) is charged less than Go's gateway-served execution, and ODE does not apply to the 0-HPK full scan — exactly where the gap is widest[4].
  • The partial-PartitionKey trap is .NET-only (Go can’t express it): a partial PartitionKeyBuilder without a WHERE clause over-reads the whole physical partition (#5404[8]). In Go the WHERE clause is the only path, which sidesteps that trap.

C · Why they differ — a source-code walkthrough

Both SDKs talk to the same service; the behavioural differences come from the client libraries, not the service itself. This section traces the source of each. Three terms first, for readers new to Cosmos: a physical partition is a backend server holding a slice of the data; the service divides the key space into partition-key ranges (one per physical partition); and a query must reach every range that could hold matching rows. Where an SDK decides which ranges to hit, and how it reads them, is the central difference between the two.

In one sentence: .NET is a thick client — it caches the partition map, plans the query, connects straight to the partition servers over TCP, and merges the results itself. Go is a thin client — it hands the SQL to the Cosmos gateway over HTTPS and lets the gateway route and execute.

C.1 · Connectivity — TCP to replicas vs HTTPS to the gateway

.NET has two modes in ConnectionMode.cs: Direct (the default) opens persistent TCP connections to the backend replicas, and Gateway routes everything through an HTTPS endpoint. Choosing Direct switches the protocol to TCP and unlocks TCP-only tuning knobs (CosmosClientOptions.cs). The Go SDK has no such choice: its client is built on a plain HTTP pipeline (azruntime.NewPipeline, cosmos_client.go) — a grep of the whole package finds no rntbd / TCP / Direct code. Go is gateway-only[3].

C.2 · Finding the partitions — a client-side map vs the gateway

.NET caches the partition layout on the client: PartitionKeyRangeCache.cs holds a CollectionRoutingMap.cs whose GetOverlappingRanges does a binary search over the ordered ranges to answer "which physical partitions overlap this key range?" — locally, with no per-query gateway call. The Go package has no routing map and no range cache; x-ms-documentdb-partitionkeyrangeid appears only as a header name constant. Go therefore never computes which partitions a query hits — it POSTs the query and the gateway decides.

C.3 · Planning the query

Before executing, .NET obtains a query plan. QueryPlanRetriever.cs offers two paths — parse it locally with the native ServiceInterop library, or fetch it from the gateway; on non-Windows platforms (Linux / macOS, as in this run) it falls back to the gateway round-trip. A fast-path, Optimistic Direct Execution (CosmosQueryExecutionContextFactory.cs), skips the plan and pipeline entirely when the query targets a single partition[4]. Go has no query-plan step at all — the SQL string goes to the gateway as-is.

C.4 · Executing across partitions — and why Go can’t aggregate

This is the central difference. .NET owns a client-side query pipeline: ParallelCrossPartitionQueryPipelineStage.cs fans out to every target partition in parallel, and merge stages sit on top — OrderBy (a k-way merge of sorted streams), Aggregate (a SingleGroupAggregator computing SUM/COUNT/AVG across partitions), and Distinct (a hash map dropping duplicates across the merged stream). Because that distribute-and-merge pipeline runs inside the client, .NET can compute cross-partition COUNT / DISTINCT / ORDER BY / GROUP BY itself. Go has none of it: cross-partition is a single header x-ms-documentdb-query-enablecrosspartition=true (defaulted on, cosmos_query_request_options.go) and the gateway does the work. The Go SDK’s own doc-comment is explicit:

"The Azure Cosmos DB Gateway API, used by the Go SDK, can only perform a LIMITED set of cross-partition queries. Specifically, the gateway can only perform simple projections and filtering on cross partition queries… If you provide a query that the gateway cannot execute, it will return a BadRequest error."— cosmos_container.go · NewQueryItemsPager[6]

That comment matches the measured result: every cross-partition COUNT failed on Go with BadRequest, while .NET returned it.

C.5 · Partial partition keys

.NET’s PartitionKeyBuilder.cs collects components via chained Add(…), and Build() makes a key with exactly as many levels as you added — so 1 or 2 of a 3-level key is a valid partial prefix, passed via QueryRequestOptions.PartitionKey. In Go, a partition_key.go PartitionKey is just a slice of values, but the query API’s doc-comment requires all levels or an empty key — "you must specify ALL partition keys that the container has" — a query-API contract, not a limit of the PartitionKey type itself. So the only route to a partial prefix in Go is the WHERE clause (§A).

C.6 · The #5404 over-read, mechanically

When a partial PartitionKey scopes a .NET query, FeedRangePartitionKey.cs turns it into an effective-partition-key (EPK) range: a full key collapses to a point, but a prefix becomes a min..max sub-range. That range routes correctly to the right physical partition — but with no WHERE predicate there is nothing index-seekable to filter within the partition, so the engine enumerates the whole scoped range (and pages fill sparsely). That is an ~11× over-read measured here (343,000 vs 31,000 docs at 2-HPK), consistent with the ~30× slowdown reported in issue #5404[8] (the exact backend read-scoping lives in the closed Microsoft.Azure.Documents layer, so this is grounded in the open source plus the issue’s own numbers). Fix: keep the WHERE clause so the index filters precisely.

1 · Prefix depth → partitions, RU, bytes (WHERE-only, SELECT *)

.NET counts distinct physical-partition GUIDs from Direct-mode store addresses (authoritative). Go counts distinct gateway x-ms-documentdb-partitionkeyrangeid headers — labelled gateway-reported. At 1-HPK WHERE SELECT * the two disagree (Go reports 3 ranges contacted, .NET reports 2 GUIDs); the .NET GUID-based count is treated as authoritative throughout this report.

2 · WHERE vs PartitionKey vs both — RU by variant (.NET, SELECT *)

Attaching a partial PartitionKeyBuilder without a WHERE clause (pk) costs substantially more RU and latency than the WHERE clause (see the ~11× over-read in §C.6) — reproduces azure-cosmos-dotnet-v3#5404. The over-read is pronounced when the prefix scopes a sub-range within a single physical partition (2-HPK here); at 1-HPK the prefix already spans whole partitions, so pk-only and WHERE are comparable (15,806 vs 16,574 RU — see Appendix A).

6 · Go — full HPK: WHERE clause vs explicit partition key vs both

For the complete 3-level key (year=2024, month=03, day=15) the Go SDK can express the same lookup three ways. This isolates whether it matters in Go — routing, RU, and latency — when the key is complete (the only case where Go can attach an explicit key).

varianthow it's written in GopartsRU (SELECT*)RU (COUNT)docsp50 @c1 (ms)p50 @c6 (ms)

pkNewPartitionKeyString("2024").AppendString("03").AppendString("15"), no WHERE1(gw)33.13.31,00069.1440.1
where+pkfull PartitionKey + full WHERE1(gw)38.83.31,00070.2475.9

Finding: with a complete key, all three route to the same single partition and cost essentially the same RU (WHERE-only 33.1 RU · explicit-PK 33.1 RU · both 38.8 RU). Specifying the key both ways tends to add a small constant overhead with no routing benefit. So for a full key the explicit partition key and the WHERE clause are interchangeable in Go — the difference that matters is only for partial prefixes, which Go can express solely through the WHERE clause.
